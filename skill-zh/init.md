# Doc Harness — 初始化

为新项目创建五个 Doc Harness 文件。

## 第一步：收集信息

需要以下信息。按优先级从三个来源获取：

**来源A — 参数**：从 `$ARGUMENTS` 中提取项目名和描述。
**来源B — 上下文**：如果对话中刚讨论了项目，从中提取目标、首要任务、约束、技术细节。
**来源C — 询问用户**：不确定的信息直接问用户。不要猜测。基于上下文的合理推断可以，凭空编造不行。

**必需信息**（创建前必须确定）：
- 项目名称（简短，用于文档标题）
- 项目描述（1-3句话）
- 第一阶段名称（初始工作叫什么）
- 第一个具体任务（先做什么——用于"车灯"部分）

**可选信息**（不确定就问，留空也可以）：
- 初始铁律
- 关键技术细节（工具、语言、路径）
- 已知子项目
- **跨项目 inbox/outbox**：问一个是/否问题——"本项目是否需要与其他项目协调（存在双向依赖）？如果是，我将启用跨项目 inbox/outbox 通信协议（spec 第14章）。" 默认：用户在上下文中提到了依赖或兄弟项目则为**是**；独立项目则为**否**。如启用，见下方第3.6步。

## 第二步：检查已有文件与项目状态

对目标目录分三种情况处理：

**(a) 干净目录**（无文件，或只有空 README/LICENSE 之类的简单起始文件）→ 进入第三步，用空白模板。

**(b) Doc Harness 已存在**（CLAUDE.md + CURRENT_STATUS.md + ... 已齐全）→ **不要覆盖**。提醒用户并建议改用 `/doc-harness check`。若只存在**部分**文件（半完成或损坏状态），应提供补齐缺失文件（即中途引入模式，见 (c)），而不是抹掉现有的部分状态。

**(c) 非空目录但无 Doc Harness**（含代码、笔记、已有 README、累积文件）→ 这是**中途引入**场景。**不要当作干净 init 处理**。在写入任何模板前：
- 阅读 `spec.md` §10.3（"适应已有项目"）了解完整流程。
- 简言之：扫描已有文件以重建项目历史；**建议**（而非强加）一个反映过去阶段的 WORKLOG 草稿；请用户确认/修正后再写入；文件较多（>~20）时按 §4.4 在 FILE_INDEX 中批量注册；按 §9.2 标注所有已被替代的文档。
- 目标是**如实描述现状**，而非假装项目刚刚开始。

**(d) 非空目录但用户明确要求全新开始**（如"忽略已有文件，我要在新任务上从零开始"）：(c) 的逃生门。需**显式**与用户确认——"已有文件**不会**被重建进 WORKLOG，它们仍在磁盘上但不会被文档系统记录。是否继续干净 init？"——用户确认后按 (a) 执行。**将用户的这项覆盖决定记入 WORKLOG 第一个阶段的总结**，以便未来 agent 理解为何没有历史重建。

## 恢复链

恢复链是任何 agent 或人类继续本项目工作的入门仪式，分两层。

### 必读（基线）

1. 本文件（CLAUDE.md）
2. `CURRENT_STATUS.md`

### 任务条件读

- 如要查找具体文件：读 `FILE_INDEX.md`
- 如要查历史阶段详情：读 `WORKLOG.md`
- [随工作类别出现添加项目特定条目]

### 元规则

- 恢复链必须**自含**：条目只指向项目内文件（或兄弟项目稳定路径）。不依赖 agent 侧特性（memory、对话历史、外部服务）。
- 恢复链是**活的**：阶段切换时审查并更新。

## 项目概述

- [项目特定规则]

## 关键技术信息

[来自上下文，或："随项目进展填充"]

## 子项目（如有）

## Doc Harness — 操作规则

[此处嵌入 operational_rules.md 的完整内容]

**重要**：读取 [operational_rules.md](operational_rules.md) 并将其完整内容（从 `<!-- doc-harness-ops-start -->` 注释到 `<!-- doc-harness-ops-end -->` 注释，**含两端**）嵌入上述 CLAUDE.md 中标记的位置。**保留两端哨兵和 `<!-- doc-harness-ops-version: N.N -->` 版本标签**——这三者界定 skill 升级后未来 `init` 重新嵌入或 `/doc-harness check` §1.10 能安全定位并替换的区域。**重新嵌入语义**：未来 `init` 或手动重新嵌入只替换 `<!-- doc-harness-ops-start -->` 与 `<!-- doc-harness-ops-end -->` 之间的字节。该区域之外的自定义铁律、项目特定章节等内容原样保留不动。

### 文件5：DOC_HARNESS_SPEC.md

读取 [spec.md](spec.md) 并将其内容写入项目目录的 `DOC_HARNESS_SPEC.md`。

### 第3.6步：（可选）启用跨项目 inbox/outbox

如用户选择启用（见第一步），执行以下操作：

**(a) 创建文件夹**：在项目根目录创建 `inbox/` 和 `outbox/`（均为空；如需 git 追踪，添加 `.gitkeep`）。

**(b) 在 CLAUDE.md 的铁律区域添加以下铁律块**（原样复制，无需修改）：

```markdown
**通过 inbox/outbox 实现跨项目通信（文件消息协议）**

- **机制**：本项目维护 `inbox/`（接收消息）和 `outbox/`（发送消息）。消息为带 YAML 前置信息的 Markdown 文件；文件名格式：`YYYY-MM-DD-from-<来源>-<主题>.md`。
- **为何必须（一旦启用）**：只在对话中交换的跨项目信息会在 session 结束时丢失。文件消息能跨越 session 边界，未来任何智能体都可通过恢复链发现并处理。
- **接收**：恢复链检查 `inbox/` 中 `status: unread` 的消息。读取后更新为 `status: read`。处理后更新为 `status: actioned`。不得修改接收消息中除 status 字段以外的任何内容；回复请写新消息。
- **发送**：写入本项目的 `outbox/`（发件方永久记录），并将同一文件复制到收件方的 `inbox/`。在 CURRENT_STATUS 中记录本次交流，以便下个 session 可追溯。
- **快照优于指针**：向其他项目传递数字/交付物时，直接在消息正文中写明数值，不要引用本项目内部文件。
- **完整规范**：本项目根目录的 `DOC_HARNESS_SPEC.md` §14。
```

**(c) 在 CLAUDE.md 恢复链的任务条件读层添加**：

```markdown
- 如 `inbox/` 中有 `status: unread` 的文件：优先读取并处理
```

**(d) 在 FILE_INDEX 中注册**：

```markdown
## 跨项目通信
- `inbox/` — 来自其他项目的接收消息
- `outbox/` — 发出的消息（草稿和发件副本）
```

## 第四步：验证

- 5个文件都存在
- FILE_INDEX 列出了全部5个文件（如已启用，还含 `inbox/` 和 `outbox/`）
- CURRENT_STATUS 有全部4个区域
- CLAUDE.md 包含嵌入的操作规则（如已启用，还含跨项目通信铁律块）
- WORKLOG 有空目录表
- 如已启用跨项目通信：`inbox/` 和 `outbox/` 存在；恢复链含任务条件读条目

报告："Doc Harness 已为 [项目名] 初始化。创建了5个文件[，跨项目 inbox/outbox 已启用（如适用）]。"
