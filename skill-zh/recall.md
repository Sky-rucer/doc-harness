# Doc Harness — Recall（信息召回）

沿项目 Doc Harness 状态文档体系检索信息。根据用户查询，在已注册文档中进行分层搜索，返回结构化、带来源引用的结果。

**触发**：用户输入 `/doc-harness recall [query]` 或自然语言等价表述。

**目的**：随着项目增长，信息分散在数十个已注册文件中。`recall` 提供一套系统化的检索协议，遵循 Recovery Chain 的层次结构——从高层面摘要到低层面细节——使 agent 能够高效找到相关信息，而无需阅读所有文件。

**与其他命令的关系**：
- `check` 审计文档**健康度**（"文档还好吗？"）
- `sync` 修复文档**漂移**（"修复文档。"）
- `flush` 将**上下文**保存到文件（"保留我们所知道的。"）
- `recall` 从文件中**检索信息**（"关于 X 我们知道什么？"）

`recall` 是**只读**的。它从不修改文件。

---

## 查询类型分类

将用户查询分类为四种类型之一。类型决定搜索哪些层级以及搜索深度。

| 类型 | 模式 | 主要层级 | 示例查询 |
|------|------|----------|----------|
| **A. 状态 / 计划** | "下一步是什么？"、"X 的当前状态？"、"计划是什么？" | 第 0–1 层 | "认证的当前计划是什么？" |
| **B. 历史 / 决策** | "我们为什么..."、"什么时候..."、"第 N 阶段"、"谁决定..." | 第 1–2 层 | "我们为什么选 PostgreSQL？" |
| **C. 文件 / 主题查找** | "找关于...的文件"、"...在哪里"、"列出...的文档" | 第 3–4 层 | "找所有关于缓存的文档" |
| **D. 跨文档综合** | "总结..."、"关于...的一切"、"所有提到...的" | 第 1–4 层 | "所有关于认证的讨论" |

**模糊查询**（不明显属于某一类型）：默认采用**类型 D**（综合）——它最全面，能从所有层级呈现信息。在输出中注明：`查询类型不明确 — 正在运行综合搜索。`

---

## 分层搜索协议

搜索按层级**自上而下**进行，严格遵守 **token 效率规则**。当查询已得到完整回答时提前停止。

**核心约束**：`recall` 不得通过阅读整个文档来消耗 token。对 FILE_INDEX 和个体文件使用定向 `Grep`，对 TOC 和特定章节使用部分 `ReadFile`，而不是全文阅读。

```
用户查询
    ↓
[阶段 1] 分类查询类型
    ↓
[阶段 2] 利用已有上下文（零 token 层）
    ↓
[阶段 3] 分层搜索（优先级顺序，token 预算控制）
    ├── 第 0 层: CLAUDE.md（铁律、概览）— 从上下文或前 20 行
    ├── 第 1 层: CURRENT_STATUS（轮胎印、车体、车灯）— 从上下文或 grep
    ├── 第 2 层: WORKLOG（TOC + 目标阶段部分）— head + grep，绝不全文阅读
    ├── 第 3 层: FILE_INDEX（仅 grep，绝不全文阅读）— grep 描述，通过 grep 递归
    └── 第 4 层: 单个文件（仅 grep 摘录，绝不全文阅读）— grep + -C 3 上下文
    ↓
[阶段 4] 综合与输出
```

### 阶段 2: 利用已有上下文（零 Token 层）

在阅读任何文件之前，检查信息是否**已经在当前上下文中**：

- 会话开始时是否已读过 CLAUDE.md 和 CURRENT_STATUS.md（恢复链）？
- 最近是否处理过与查询主题相关的文件？
- 答案是否已从本对话的前几轮中得知？

如果是 → 直接使用该信息。不要重新阅读你已知道的文件。

**这是最重要的省 token 步骤。** 大多数 recall 查询可以通过上下文加一次快速 grep 来回答。

### 第 0 层: CLAUDE.md

**何时阅读**：如果铁律或恢复链相关且不在上下文中。

**如何阅读**：**只读前 20 行**（单行状态、当前阶段、铁律标题）。除非查询专门关于恢复链结构，否则不要读完整文件。

### 第 1 层: CURRENT_STATUS.md

**何时阅读**：如果不在上下文中，或自上次阅读以来车体已增长。

**如何阅读**：使用 `Grep` 配合查询关键词，或按标题读取特定部分（例如，类型 A 只读 `## Next Steps`）。不要从头到尾读完整文件。

### 第 2 层: WORKLOG.md

**何时阅读**：用于类型 B（历史/决策）和类型 D（综合）。纯类型 A 跳过。

**如何阅读（严格规则）**：
1. **只读 TOC**（前 20 行）以识别阶段名称和锚点
2. **Grep** 查询关键词贯穿整个 WORKLOG —— 这告诉你哪些阶段提到了该主题
3. **只读匹配的阶段部分**（不是整个 WORKLOG）。使用 grep 结果定位正确的 `##` 标题，然后读取该标题周围的 20–50 行。

**绝不全文阅读 WORKLOG** —— 它可能有 500+ 行。

### 第 3 层: FILE_INDEX + 子索引

**何时阅读**：用于类型 C 和类型 D。类型 A/B 除非第 1–2 层不足，否则跳过。

**如何阅读（严格规则）**：
1. **仅 grep** —— 在 FILE_INDEX.md 上使用 `Grep` 配合查询关键词。**不要**读完整 FILE_INDEX。
2. **记录匹配条目**（路径 + 描述）
3. **子索引递归**：如果匹配条目指向 sub-FILE_INDEX.md，**也 grep 那个子索引** —— 不要全文阅读。

**为什么不全文阅读**：FILE_INDEX 可能有 100+ 行。Grep 可以在一次工具调用中给你精确的匹配。

### 第 4 层: 单个文件

**何时阅读**：当第 1–3 层识别出相关文件且描述不足时。

**如何阅读（严格规则）**：
1. **先 grep** —— 在每个候选文件上运行 `Grep` 配合查询关键词。使用 `-C 3`（3 行上下文）来捕获 surrounding 段落。
2. **仅当 grep 没找到但文件被强烈怀疑时** → 读前 30 行（概览/TOC）看看主题是否在用不同术语讨论。
3. **绝不全文阅读文件**除非它 <50 行且你有强证据它包含答案。

**范围限制**：如果第 3 层有 >10 个文件匹配：
- 呈现前 5 个最相关的匹配（按描述相关性）
- 告诉用户："5+ 个文件匹配。显示前 5 个。指定类别或重新表述以缩小范围。"
- 未经用户确认，不要在第 4 层阅读超过 5 个文件。

## 搜索规则（规范性）

1. **只搜索已注册文件**（在 FILE_INDEX 或子索引中）。未注册文件对 `recall` 不可见。
2. **上下文优先，文件其次**：在阅读任何内容之前，始终检查答案是否已在对话上下文中。
3. **先 grep 再阅读**：对 FILE_INDEX 和个体文件，使用 `Grep` 作为第一个工具调用。全文 `Read` 是最后手段。
4. **绝不全文阅读大文件**：
   - WORKLOG >200 行 → grep + 只读匹配部分
   - FILE_INDEX >50 行 → 仅 grep
   - 任何个体文件 >100 行 → grep + -C 3 上下文
5. **通过 grep 处理子索引**：使用 grep 递归进入 sub-FILE_INDEX.md，不要全文阅读。
6. **引用每个声明**：每个事实必须携带引用如 `(CURRENT_STATUS.md §车体)` 或 `(notes/design.md#L42)`。
7. **"未找到" 是可接受的**：清楚说明。不要幻觉。
8. **不修改文件**：`recall` 是只读的。注明漂移但不修复。
9. **提前停止**：如果第 1–2 层已经回答了查询，不要进入第 4 层。

## 输出格式

```
═══════════════════════════════════════
  Doc Harness — Recall
  查询: [用户的精确查询]
  类型: [A | B | C | D | 不明确→D]
  搜索层级: [0–1 | 0–2 | 0–3 | 0–4]
═══════════════════════════════════════

── 第 0 层: 项目概览 ──
[来自 CLAUDE.md 的相关摘录，如果有]

── 第 1 层: 当前状态 ──
[来自 CURRENT_STATUS 的相关摘录，带引用]

── 第 2 层: 工作日志历史 ──
[来自 WORKLOG 的相关阶段摘要，带引用]
（如果跳过则省略）

── 第 3 层: 文件索引匹配 ──
[匹配的 FILE_INDEX 条目及其描述]
（如果没有匹配或跳过则省略）

── 第 4 层: 文档详情 ──
[如果阅读了：来自单个文件的关键摘录，带引用]
（如果跳过则省略）

── 综合 ──
[对用户问题的简洁回答；每个声明都有引用]

── 未找到 ──
[如果适用："以下主题在已注册文档中未找到: ..."]

── 漂移备注 ──
[如果适用："注意：recall 发现以下未注册文件可能相关: ..."]
═══════════════════════════════════════
```

**省略规则**：如果某层被跳过（因为较早的层已完全回答查询），完全省略该层的部分。不要写 "第 2 层: (已跳过)。"

## 边界情况

### 查询无匹配

```
── 综合 ──
没有已注册文档提及 "[查询主题]"。

可能原因：
- 该主题尚未被记录。
- 相关文件存在但未在 FILE_INDEX 中注册。
- 该主题使用了不同的术语描述。

建议：运行 `/doc-harness sync` 确保所有文件都已注册，然后重试。
```

### 查询匹配未注册文件

如果对整个文件系统的 grep 在**未注册**文件中发现匹配，在 `漂移备注` 部分注明它们，但不要阅读其内容（它们不属于 Doc Harness 信任边界）：

```
── 漂移备注 ──
以下未注册文件也提到了 "[主题]" 但未被阅读：
- `tmp/draft.md`
- `data/experiment_notes.txt`

如果它们包含持久信息，考虑在 FILE_INDEX 中注册它们。
```

### 非常大的项目 (>100 个已注册文件)

如果 FILE_INDEX 列出 100+ 个文件（包括子索引）：
1. 先阅读 FILE_INDEX 类别
2. 只搜索描述与查询匹配的类别
3. 对于匹配类别中有 >20 个文件的，呈现类别列表并询问用户深入阅读哪些
4. 单次 recall 中永远不要阅读超过 20 个单个文件

## 何时使用

- 用户问 "我们为什么决定 X？" → 类型 B
- 用户问 "Y 的当前状态是什么？" → 类型 A
- 用户问 "找所有关于 Z 的文件" → 类型 C
- 用户问 "总结关于 W 的一切我们知道的" → 类型 D
- 用户问模糊的开放式问题，暗示 "搜索项目文档"

## 何时不使用

- 当用户明确要求你**创建**某物时——那是正常工作，不是检索。
- 当问题可以仅从上下文中回答时（例如，"你刚才说了什么？"）。
- 当用户明确要求你**修改**文件时——检索是副作用，不是目标。
