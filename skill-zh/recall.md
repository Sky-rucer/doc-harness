# Doc Harness — Recall（信息召回）

沿项目 Doc Harness 状态文档体系检索信息。根据用户查询，在已注册文档中进行分层搜索，返回结构化、带来源引用的结果。

**触发**：用户输入 `/doc-harness recall [query]` 或自然语言等价表述。

**目的**：随着项目增长，信息分散在数十个已注册文件中。`recall` 提供一套系统化的检索协议，遵循 Recovery Chain 的层次结构——从高层面摘要到低层面细节——使 agent 能够高效找到相关信息，而无需阅读所有文件。

**与其他命令的关系**：
- `check` 审计文档**健康度**（"文档还好吗？"）
- `sync` 修复文档**漂移**（"修复文档。"）
- `flush` 将**上下文**保存到文件（"保留我们所知道的。"）
- `recall` 从文件中**检索信息**（"关于 X 我们知道什么？"）

`recall` 是**只读**的。它从不修改文件。

---

## 查询类型分类

将用户查询分类为四种类型之一。类型决定搜索哪些层级以及搜索深度。

| 类型 | 模式 | 主要层级 | 示例查询 |
|------|------|----------|----------|
| **A. 状态 / 计划** | "下一步是什么？"、"X 的当前状态？"、"计划是什么？" | 第 0–1 层 | "认证的当前计划是什么？" |
| **B. 历史 / 决策** | "我们为什么..."、"什么时候..."、"第 N 阶段"、"谁决定..." | 第 1–2 层 | "我们为什么选 PostgreSQL？" |
| **C. 文件 / 主题查找** | "找关于...的文件"、"...在哪里"、"列出...的文档" | 第 3–4 层 | "找所有关于缓存的文档" |
| **D. 跨文档综合** | "总结..."、"关于...的一切"、"所有提到...的" | 第 1–4 层 | "所有关于认证的讨论" |

**模糊查询**（不明显属于某一类型）：默认采用**类型 D**（综合）——它最全面，能从所有层级呈现信息。在输出中注明：`查询类型不明确 — 正在运行综合搜索。`

---

## 分层搜索协议

搜索按层级**自上而下**进行。当查询已得到完整回答时提前停止——如果第 1–2 层已提供完整答案，不要强制阅读第 4 层文件。

```
用户查询
    ↓
[阶段 1] 分类查询类型
    ↓
[阶段 2] 分层搜索（优先级顺序）
    ├── 第 0 层: CLAUDE.md（铁律、概览、恢复链）
    ├── 第 1 层: CURRENT_STATUS（轮胎印、车体、车灯、驾驶手册）
    ├── 第 2 层: WORKLOG（历史阶段摘要）
    ├── 第 3 层: FILE_INDEX + 子索引（目录扫描）
    └── 第 4 层: 单个已注册文件（需要时全文搜索）
    ↓
[阶段 3] 综合与输出
```

### 第 0 层: CLAUDE.md

**何时阅读**：总是先读（它很小）。提取：
- 可能约束答案的铁律
- 项目概览以提供上下文
- 可能相关的恢复链条目

**停止条件**：如果查询关于铁律或恢复链，在此层后回答。

### 第 1 层: CURRENT_STATUS.md

**何时阅读**：总是阅读（它很小，约 100–200 行）。

按顺序搜索以下部分：
1. **车灯** (`## Next Steps`) — 用于类型 A（状态/计划）查询
2. **车体** (`## Current Work`) — 用于类型 A 和 B
3. **轮胎印** (`## Recent Completed`) — 用于类型 B（历史）查询
4. **驾驶手册** (`## Current Working Principles`) — 用于规则相关查询

**停止条件**：如果 CURRENT_STATUS 完全回答了查询（例如，"下一步是什么？" → 车灯），在此停止。

### 第 2 层: WORKLOG.md

**何时阅读**：用于类型 B（历史/决策）和类型 D（综合）。对于纯类型 A（状态/计划），除非用户明确要求历史对比，否则跳过。

**搜索策略**：
1. 阅读 TOC（前 20 行）以识别相关阶段
2. 通过锚点链接跳转到匹配的阶段部分
3. 提取提及查询主题的阶段摘要

**停止条件**：如果 WORKLOG 完全回答了历史问题，在此停止。

### 第 3 层: FILE_INDEX + 子索引

**何时阅读**：用于类型 C（查找）和类型 D（综合）。对于类型 A/B，仅当第 1–2 层不足时才阅读。

**搜索策略**：
1. 阅读根 FILE_INDEX.md
2. 对每个类别，在条目描述中 grep 查询关键词
3. 如果某个类别指向 sub-FILE_INDEX.md，递归进入
4. 记录匹配条目及其单行描述

**子索引递归规则**：当 FILE_INDEX 包含如下条目时：
```
- `notes/findings/` → 见 [`notes/findings/FILE_INDEX.md`]
```
阅读该 sub-FILE_INDEX 并同样搜索。

**停止条件**：对于类型 C，如果用户只想要文件列表，收集匹配后停止。

### 第 4 层: 单个文件

**何时阅读**：当第 1–3 层识别出相关文件，但其描述不足以回答查询时。

**搜索策略**：
1. 优先阅读第 3 层匹配的文件
2. 阅读每个文件的全文（或 grep 查询关键词）
3. 提取带行号/章节引用的相关摘录

**范围限制**：如果第 3 层有 >20 个文件匹配，**不要全部阅读**。而是：
- 向用户呈现 FILE_INDEX 匹配摘要
- 询问："20+ 个文件匹配。我应该深入阅读哪些类别？"
- 或者，如果查询足够窄，只阅读前 5 个最相关的文件，并注明："显示前 5 个匹配 — 还有更多。"

## 搜索规则（规范性）

1. **只搜索已注册文件**（在 FILE_INDEX 或子索引中）。未注册文件对 `recall` 不可见——这激励遵守核心原则进行注册。
2. **尊重恢复链优先级**：CURRENT_STATUS 中的摘要优先于原始文件。如果轮胎印已经回答了问题，不要强制阅读 10 个源文件。
3. **子索引是一等公民**：像 FILE_INDEX 遍历一样递归进入 sub-FILE_INDEX.md 文件。
4. **引用每个声明**：每个事实必须携带引用：
   - `(CLAUDE.md §铁律)`
   - `(CURRENT_STATUS.md §车体)`
   - `(WORKLOG.md §第 3 阶段)`
   - `(FILE_INDEX.md → notes/design.md)`
   - `(notes/design.md §数据库选择, L42)`
5. **"未找到" 是可接受的**：如果查询没有匹配，清楚说明。不要幻觉或猜测。
6. **不修改文件**：`recall` 是只读的。如果你在 recall 过程中注意到漂移，在输出中注明但不要修复——那是 `sync` 的工作。
7. **提前停止**：如果第 1–2 层已经回答了查询，不要阅读第 4 层。

## 输出格式

```
═══════════════════════════════════════
  Doc Harness — Recall
  查询: [用户的精确查询]
  类型: [A | B | C | D | 不明确→D]
  搜索层级: [0–1 | 0–2 | 0–3 | 0–4]
═══════════════════════════════════════

── 第 0 层: 项目概览 ──
[来自 CLAUDE.md 的相关摘录，如果有]

── 第 1 层: 当前状态 ──
[来自 CURRENT_STATUS 的相关摘录，带引用]

── 第 2 层: 工作日志历史 ──
[来自 WORKLOG 的相关阶段摘要，带引用]
（如果跳过则省略）

── 第 3 层: 文件索引匹配 ──
[匹配的 FILE_INDEX 条目及其描述]
（如果没有匹配或跳过则省略）

── 第 4 层: 文档详情 ──
[如果阅读了：来自单个文件的关键摘录，带引用]
（如果跳过则省略）

── 综合 ──
[对用户问题的简洁回答；每个声明都有引用]

── 未找到 ──
[如果适用："以下主题在已注册文档中未找到: ..."]

── 漂移备注 ──
[如果适用："注意：recall 发现以下未注册文件可能相关: ..."]
═══════════════════════════════════════
```

**省略规则**：如果某层被跳过（因为较早的层已完全回答查询），完全省略该层的部分。不要写 "第 2 层: (已跳过)。"

## 边界情况

### 查询无匹配

```
── 综合 ──
没有已注册文档提及 "[查询主题]"。

可能原因：
- 该主题尚未被记录。
- 相关文件存在但未在 FILE_INDEX 中注册。
- 该主题使用了不同的术语描述。

建议：运行 `/doc-harness sync` 确保所有文件都已注册，然后重试。
```

### 查询匹配未注册文件

如果对整个文件系统的 grep 在**未注册**文件中发现匹配，在 `漂移备注` 部分注明它们，但不要阅读其内容（它们不属于 Doc Harness 信任边界）：

```
── 漂移备注 ──
以下未注册文件也提到了 "[主题]" 但未被阅读：
- `tmp/draft.md`
- `data/experiment_notes.txt`

如果它们包含持久信息，考虑在 FILE_INDEX 中注册它们。
```

### 非常大的项目 (>100 个已注册文件)

如果 FILE_INDEX 列出 100+ 个文件（包括子索引）：
1. 先阅读 FILE_INDEX 类别
2. 只搜索描述与查询匹配的类别
3. 对于匹配类别中有 >20 个文件的，呈现类别列表并询问用户深入阅读哪些
4. 单次 recall 中永远不要阅读超过 20 个单个文件

---

## 何时使用

- 用户问 "我们为什么决定 X？" → 类型 B
- 用户问 "Y 的当前状态是什么？" → 类型 A
- 用户问 "找所有关于 Z 的文件" → 类型 C
- 用户问 "总结关于 W 的一切我们知道的" → 类型 D
- 用户问模糊的开放式问题，暗示 "搜索项目文档"

## 何时不使用

- 当用户明确要求你**创建**某物时——那是正常工作，不是检索。
- 当问题可以仅从上下文中回答时（例如，"你刚才说了什么？"）。
- 当用户明确要求你**修改**文件时——检索是副作用，不是目标。
